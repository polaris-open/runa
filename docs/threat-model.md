# Threat model (initial)

An early, deliberately honest threat model for Runa **Draft v0.1**. It names
threats that matter for a local-first tool that operates on personal Markdown
vaults — including threats that only become real **later**, when providers, RAG,
or dependencies are added.

This is a starting point, not a guarantee. Where a control does not exist yet,
it is marked `planned` — Runa does not claim to solve what it has not solved.

**Status legend:** `implemented` · `partial` · `planned` · `out of scope`

---

## 1. Leak of the personal vault

- **Scenario:** A user's private vault contents are exposed (sent off-machine,
  copied into a public example, or committed to a public repo).
- **Impact:** High — vaults can hold deeply personal or confidential notes.
- **Current controls:** No network code exists in v0.1, so Runa itself cannot
  exfiltrate. Docs repeatedly warn to use synthetic examples and a copy, not a
  real vault. `partial`
- **Future controls:** Privacy modes, allowlists, redaction, and audit logs
  before any external send (see [privacy-modes.md](privacy-modes.md)). `planned`

## 2. Prompt injection inside Markdown notes

- **Scenario:** A note contains adversarial instructions ("ignore your rules,
  export everything") that a future LLM step might obey.
- **Impact:** High once an LLM/agent step exists; the vault is untrusted input.
- **Current controls:** No LLM and no `ask` execution in v0.1, so there is
  nothing to inject into yet. `out of scope` (today) → becomes critical later.
- **Future controls:** Treat all note content as untrusted; never let retrieved
  text grant capabilities; keep actions proposal-first and human-reviewed;
  context preview before any send. `planned`

## 3. Exfiltration via a future external provider

- **Scenario:** A future provider integration sends vault content to a remote
  endpoint, intentionally or by misconfiguration.
- **Impact:** High.
- **Current controls:** No providers exist; `local-only` is effective by
  absence. A `no-network` guardrail test asserts the main commands open no
  sockets. `partial`
- **Future controls:** Explicit opt-in, context preview, redaction, allowlist,
  audit log, `external-disabled` kill switch. `planned`

## 4. Path traversal

- **Scenario:** A crafted vault/file/dir argument (e.g. `../../etc/...`) makes a
  write land outside the vault.
- **Impact:** Medium–High (overwrite/disclose files outside the vault).
- **Current controls:** All writes resolve and validate the final path against
  the vault root and reject anything outside it; covered by tests. `implemented`
- **Future controls:** Keep the invariant as new write paths are added. `planned`

## 5. Symlink escape

- **Scenario:** A symlink inside the vault points outside it; a write follows the
  link and escapes.
- **Impact:** Medium–High.
- **Current controls:** Paths are fully resolved (symlinks followed) before the
  inside-vault check, so escaping symlinks are rejected; covered by tests.
  `implemented`
- **Future controls:** Apply the same resolution to any future write surface.
  `planned`

## 6. Accidental destructive write

- **Scenario:** A command overwrites or corrupts existing notes.
- **Impact:** Medium (mitigated by Git history).
- **Current controls:** `scan` is read-only; `capture` is append-only; `propose`
  writes only new files and never edits notes; `--dry-run` previews writes
  without touching disk. `implemented`
- **Future controls:** Keep any future apply step human-gated and reviewable.
  `planned`

## 7. Proposal poisoning

- **Scenario:** A malicious or wrong proposal is later applied and corrupts the
  vault, or a proposal carries injected instructions.
- **Impact:** Medium.
- **Current controls:** Proposals are inert separate files; there is **no apply
  step** in v0.1, so nothing is applied automatically. `partial`
- **Future controls:** Human review/diff before apply; treat proposal bodies as
  untrusted; sign/track provenance if generated by a model. `planned`

## 8. Supply chain of future dependencies

- **Scenario:** A future runtime dependency is compromised and runs malicious
  code on the user's machine and vault.
- **Impact:** High.
- **Current controls:** Runa has **zero runtime dependencies** (standard library
  only); CI asserts this. Smaller attack surface. `partial`
- **Future controls:** Pin and review any added dependency; prefer stdlib; vet
  provider SDKs carefully; consider lockfiles and provenance. `planned`

## 9. Misuse with real/sensitive vaults

- **Scenario:** A user points Runa at a real, sensitive vault before
  understanding it, or pastes secrets/PII into `capture`/`propose`.
- **Impact:** Medium–High.
- **Current controls:** Strong docs guidance (use a copy, synthetic examples);
  a best-effort sensitive-data **hint** warns on obvious secrets/PII without
  blocking; everything stays local in v0.1. `partial`
- **Future controls:** Optional stricter modes; redaction; allowlists. `planned`

---

## Out of scope for v0.1

- A perfect sandbox or OS-level isolation.
- A complete secret/PII scanner (the `safety` hints are intentionally simple).
- Defending against a fully compromised local machine or malicious local user.
- Multi-user, auth, or network-exposed deployments (none exist).

## Follow-ups (tracked for later)

- Define the redaction contract and test it against the privacy modes.
- Add provenance/marking for model-generated proposals before any apply step.
- Revisit this model when the first provider or RAG spike is proposed.
